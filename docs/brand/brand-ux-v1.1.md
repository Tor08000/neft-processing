# Brand UX v1.1 — Закрепление

## Цель
Сделать NEFT предсказуемым, спокойным и дисциплинирующим интерфейсом.
Не умнее пользователя. Не громче проблемы. Не эмоциональнее факта.

## ШАГ 1. Brand UX Audit

Принцип: **1 экран = 1 вердикт**

### 1. Admin Explain (Explain v2 / v2.1)

**Вердикт:** ⚠️ перегружено

**Почему:**
- Много уровней объяснения на одном экране.
- Пользователь сразу видит:
  - дерево причин
  - evidence
  - действия
  - diff / snapshot
- Это логично для системы, но психологически тяжело для человека.

**Что хорошо (оставляем):**
- Иерархия «причина → следствие → действие»
- Отсутствие паники и красных алармов
- “Почему?” вместо “Ошибка”

**Что упрощать:**
- Один основной ответ наверху (1–2 строки)
- Остальное — свернуть по умолчанию
- Diff / Snapshot → только по явному действию пользователя

**Итог:**
Explain должен сначала успокаивать, потом объяснять, и только потом позволять углубиться.

### 2. Client Explain

**Вердикт:** ✅ соответствует

**Почему:**
- Чёткий, ограниченный доступ
- Нет ощущения «меня проверяют»
- Нет операторского перегруза
- Есть ощущение: «система меня защищает»

**Рекомендация:**
- Сохранить жёсткое ограничение глубины
- Никаких “Diff”, “What-if”, “Snapshot” без роли

### 3. Case list / Case details

**Вердикт:** ⚠️ слишком “тикетно”

**Почему:**
- Список кейсов визуально напоминает support/CRM
- Статусы читаются как “обработан / не обработан”
- Теряется ощущение: это не жалоба, это разбор ситуации

**Что хорошо:**
- Таймлайн
- Системные комментарии
- SLA как факт, а не угроза

**Что менять:**
- Убрать “тикетный” язык:
  - ❌ «Открыт / Закрыт»
  - ✅ «Активен / Завершён»
- Убрать ощущение “работы с клиентом”
- Подчеркнуть: Case = внутренняя дисциплина системы

### 4. Escalation states

**Вердикт:** ✅ соответствует

**Почему:**
- Escalation не выглядит аварией
- Нет истерики
- Очереди воспринимаются как роли, а не «отделы тушения пожаров»

**Важно сохранить:**
- Escalation = процесс, а не наказание
- SLA = таймер дисциплины, не давления

## ШАГ 2. Brand UX Guardrails

(мини-гайд для Codex и всех изменений дальше)

### 1. Язык (Copy)

**Можно:**
- «Система обнаружила…»
- «Причина: …»
- «Требует внимания»
- «Рекомендуемое действие»

**Нельзя:**
- «Ошибка»
- «Критично»
- «Проблема»
- «Срочно!»
- «Нарушение!»

NEFT не пугает. NEFT фиксирует факт.

### 2. Кнопки

**Разрешённые форматы:**
- Посмотреть объяснение
- Открыть кейс
- Применить действие
- Сравнить
- Вернуться

**Запрещено:**
- Исправить
- Решить
- Устранить
- Починить

NEFT не «чинит» — она управляет.

### 3. Цвета (критично важно)

**Зелёный:**
- Норма
- Завершено
- В пределах SLA

**Жёлтый:**
- Внимание
- Ожидание
- Требуется решение

**Красный — ЗАПРЕТ ПО УМОЛЧАНИЮ**

Красный нельзя использовать, если:
- система ещё контролирует ситуацию
- есть доступное действие
- нет потерь данных/денег

Красный допустим только:
- при необратимом факте
- при юридическом/финансовом событии
- при внешнем отказе без fallback

### 4. Empty / Error / Loading — единый язык

**Loading:**
«Система анализирует данные…»

**Empty:**
«Событий не обнаружено за выбранный период.»

**Error:**
«Данные временно недоступны. Система продолжает мониторинг.»

**Никогда:**
- «Что-то пошло не так»
- «Ошибка загрузки»

### 5. Иерархия экранов (золотое правило)

1. Факт
2. Причина
3. Статус
4. Действие (если доступно)
5. Детали (по запросу)

Если порядок нарушен — UX считается сломанным.
