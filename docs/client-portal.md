# NEFT Client Portal (client-web)

Первая версия клиентского кабинета размещается как отдельное фронтенд-приложение `frontends/client-portal` и
отдаётся через gateway по базовому префиксу `/client/`. Приложение использует только клиентские API и не
пересекается с админской частью.

## Разделение порталов и безопасность

Client Portal — отдельный UI только для клиентов (B2B/B2C) и их сотрудников/водителей. Он не пересекается
с Partner Portal и Admin UI (NEFT Ops). Разрешены только токены с `issuer`/`audience`, предназначенными для
клиентской зоны (например, `issuer=neft-client`), чтобы исключить попадание `superadmin/support/risk/finance`
в клиентский кабинет.

## Сервисы и маршрутизация

- **Static:** gateway проксирует `/client/` и `/client/assets/` в upstream `client-web`.
- **API:** клиентские запросы проходят через gateway по префиксу `/client/api/v1/...`:
  - `/client/api/v1/auth/*` → `auth-host` (логин, `GET /auth/me` для клиентских пользователей);
  - `/client/api/v1/*` → `core-api` (операции, лимиты, дашборд и остальные ручки).

## API клиента

Минимальный набор эндпоинтов для фронта:

- `POST /client/api/v1/auth/login` — вход по email/паролю с указанием роли и состояния организации.
- `GET /client/api/v1/auth/me` — информация о текущем клиентском пользователе и его организации.
- `GET /client/api/v1/operations` — список операций клиента с фильтрами по датам, типу и статусу.
- `GET /client/api/v1/limits` — активные лимиты организации/карт с остатком и периодом.
- `GET /client/api/v1/dashboard` — агрегированные данные для первого экрана (кол-во операций, оборот,
  активные лимиты/предупреждения).

В репозитории реализована демо-версия этих эндпоинтов: `auth-host` обслуживает `/client/api/v1/auth/*`,
а `core-api` — `/client/api/v1/operations`, `/client/api/v1/limits` и `/client/api/v1/dashboard`. Через
gateway все запросы доступны по тем же путям с префиксом `/client/`.

Авторизация должна выполняться через `auth-host`, токен содержит тип субъекта (`client_user`), роли клиента
и идентификатор организации, что используется для фильтрации данных на backend (RBAC + ABAC).

## Клиентские роли (RBAC)

Роли существуют только внутри компании клиента и не пересекаются с платформенными ролями:

- **OWNER** — владелец компании, полный доступ в рамках своей организации.
- **ADMIN** — управление пользователями, картами и лимитами.
- **ACCOUNTANT** — документы/счета/акты/выгрузки.
- **FLEET_MANAGER** — автопарк/водители/карты/лимиты по ТС.
- **LOGISTICIAN** — рейсы/маршруты/контроль расхода (если модуль включён).
- **DRIVER** — доступ только к своим картам/операциям/лимитам.

## Company Shell и онбординг компании

Онбординг компании выполняется внутри клиентского кабинета после входа:

1. Регистрация аккаунта (email/пароль/OTP).
2. Вход в Client Portal.
3. Экран «Компания не подключена».
4. Wizard подключения компании:
   - профиль компании (ИНН/КПП/адрес);
   - договор (просмотр/подпись ПЭП);
   - выбор подписки (например: FREE/CONTROL/OPTIMIZE/ENTERPRISE).
5. После активации доступны базовые разделы: **Карты**, **Пользователи**, **Документы**.

## Минимальные разделы Client Portal (v1)

1. **Подписка** — текущий план, доступные планы, модули, ограничения (cards/users/vehicles).
2. **Карты** — список, выпуск (если разрешено), статусы, лимиты, привязки (водитель/ТС/подразделение),
   история операций по карте.
3. **Сотрудники/водители** — приглашения, роли, доступ к ресурсам (карты/документы/подразделения).
4. **Документы и договоры** — договор, счета, акты, закрывающие документы.
5. **Marketplace** — витрина/заказы/документы (если модуль включён).

## ABAC: доступ водителей к картам

Для выдачи доступа к 1..N картам используется сущность привязки:

**card_access**

- `user_id`
- `card_id`
- `scope` (READ/USE/MANAGE)
- `effective_from` / `effective_to`
- `created_by`

Это позволяет назначать временный доступ, делегировать права и формально ограничивать видимость карт.

## Пользовательский поток

1. Пользователь открывает `/client/` и видит форму логина.
2. После успешного входа:
   - если компания не подключена — отображается Company Shell и wizard подключения;
   - если подключена — отображается шапка с названием организации и ролью пользователя.
3. Боковое меню ведёт к базовым страницам (Карты, Пользователи, Документы) и
   дополнительным разделам в зависимости от подписки.
4. Все запросы идут к `/client/api/v1/...` с токеном авторизации (issuer для client-portal).

## Локальный запуск

```bash
# Собрать и запустить весь стек c клиентским фронтом
docker compose up --build gateway client-web

# Открыть клиентский кабинет
open http://localhost/client/
```

При использовании Vite-приложения напрямую:

```bash
cd frontends/client-portal
npm install
npm run dev -- --host --port 4174
# браузер: http://localhost:4174/client/
```
