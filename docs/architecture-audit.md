# Архитектурный аудит NEFT Processing

## Методология
- Быстрый обзор кода сервисов (`core-api`, `auth-host`, `workers`, `gateway`, `admin-web`) и инфраструктуры (`docker-compose.yml`, Nginx-конфиг, схемы БД).
- Фокус на API-границах, данных, очередях и фронтенде.
- Приоритеты: **P1 — критично/блокеры**, **P2 — важно**, **P3 — желательно**.

## Ключевые наблюдения

### API (FastAPI / core-api)
- В одном приложении смешаны публичные и админские эндпоинты; admin-ручки подключаются напрямую в `main.py` без отдельного префикса/скоупа (мешает политике RBAC и rate-limit).【F:services/core-api/app/main.py†L15-L55】
- Версионирование используется частично: `api/v1` для основных роутов, но админ-часть подключается без явного пространства имен и может конфликтовать при будущих версиях.【F:services/core-api/app/main.py†L15-L55】
- Pydantic-модели объявлены локально в `main.py`, дублируя данные домена и не переиспользуются между сервисами (риск рассинхрона и расходимостей).【F:services/core-api/app/main.py†L62-L115】

### Postgres
- Таблица `operations` растёт и использует множество одиночных B-Tree индексов; нет составных индексов по частым фильтрам (например, `(merchant_id, created_at)`), нет FK на справочники терминалов/клиентов, что упрощает вставку, но усложняет целостность и планы запросов.【F:services/core-api/db/models/operation.py†L14-L137】
- В `docker-compose` отсутствуют параметры по лимитам памяти/автовакуума; PGOFF/`shared_buffers` не управляются, что может ухудшить производительность при нагрузке.【F:docker-compose.yml†L2-L198】

### Redis
- Redis запускается в режиме AOF, но без настроек TTL/eviction и без разделения БД под брокер/result backend/кеш; Celery уже занимает DB0/DB1, но остальные данные могут конкурировать за память и ключи.【F:docker-compose.yml†L19-L165】
- Нет мониторинга TTL или метрик hit/miss; сложно понять эффективность кеширования.

### Celery / workers
- Брокер и backend в общем Redis без квот; отсутствуют настройки max_tasks_per_child, prefetch, rate-limit задач или политики ретраев в коде (общая точка риска для дедлоков и проглатывания сообщений).【F:docker-compose.yml†L119-L173】
- Flower поднят, но без аутентификации/ограничений доступа, что создаёт security-risk и позволяет менять статус задач извне.【F:docker-compose.yml†L154-L175】

### Gateway (Nginx)
- Используется HTTP/1.1, нет gzip/br/HTTP2, нет cache-control для статических файлов; каждая SPA-страница идёт без компрессии и кеширования.【F:services/gateway/nginx.conf†L17-L72】
- Нет circuit-breaking/timeout настроек для бэкендов, что может вешать клиентов при деградации.

### Admin-web (Vite + React)
- Сборка без code splitting и без статических хедеров кеширования на уровне gateway (Nginx конфиг отдаёт как есть).【F:services/admin-web/package.json†L1-L22】【F:services/gateway/nginx.conf†L48-L72】
- Нет анализа bundle size/ESM chunking; отсутствие API-клиента с кешированием приводит к повторным запросам.

## План улучшений

### P1 — критично
1. **Разделить публичные и админские API**
   - Вынести админ-роуты в отдельный роутер с префиксом `/api/v1/admin` и отдельной зависимостью авторизации/роли; задокументировать в OpenAPI.
2. **Усилить безопасность очередей**
   - Включить аутентификацию для Flower или ограничить доступ сетевыми правилами; добавить параметры retry/backoff по умолчанию в Celery-тасках и контролировать prefetch.
3. **Оптимизация таблицы operations**
   - Ввести FK на merchants/terminals/clients (если справочники есть) или хотя бы check constraints; добавить составные индексы по `(merchant_id, created_at)`, `(terminal_id, status, created_at)` и partial index по открытым операциям.

### P2 — важно
1. **Версионирование и схемы**
   - Перевести все эндпоинты (включая админские) под `/api/v1/...`; вынести Pydantic-схемы в общий модуль/пакет `schemas`, генерировать их из доменных моделей, чтобы избежать дубликатов.
2. **Redis как кеш/брокер**
   - Разнести брокер/результаты Celery и кеш в отдельные DB/инстансы; включить expiry для временных ключей, включить eviction policy (`allkeys-lru`) для кеша.
3. **Настройки Postgres**
   - Добавить `maintenance_work_mem`, `max_wal_size`, `autovacuum_naptime` через `postgresql.conf` или docker env; включить мониторинг VACUUM/ANALYZE.
4. **Gateway производительность**
   - Включить gzip/brotli, HTTP/2, add_header Cache-Control для статических ресурсов, таймауты `proxy_read_timeout`, `proxy_connect_timeout`.

### P3 — желательно
1. **Админ SPA**
   - Подключить code splitting (dynamic import) и prefetch для крупных страниц; добавить сервис-воркер или HTTP кеширование для статических чанков.
   - Ввести клиентский кеш (React Query/SWR) для повторяющихся запросов журнала операций.
2. **Наблюдаемость Celery**
   - Метрики Prometheus для длительности задач, глубины очередей; алерты на рост очередей и количество ретраев.
3. **Документация и тесты API**
   - Автотесты контрактов (schemathesis/pact) для public/admin API; регламент по миграциям БД и индексации больших таблиц.

## Итог
- Сильные стороны: единый docker-compose для всего стека, базовые healthchecks, разделение сервисов (auth/core/ai/workers/gateway).
- Основные риски: отсутствие чёткой границы admin/public API, недостаточная оптимизация БД и очередей, слабая фронтенд-оптимизация и сетевая компрессия.
- Предложенные шаги позволяют повысить производительность, безопасность и управляемость системы с минимальными изменениями в архитектуре.
