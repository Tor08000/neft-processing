## Pre-External Exclusions

These production/provider or write contours stay outside the internal release gate on purpose and must not be described as green-by-default:

- marketplace recommendation tails without mounted owner proof
- partner contract/settlement write and approval tails
- production provider-backed OTP, email/SMS delivery, live EDO transport, bank API, ERP/1C sync, and external fuel/logistics credentials

Gate meaning:

- mounted internal owners above must be `VERIFIED_RUNTIME` or `VERIFIED_SKIP_OK`
- optional disabled contours may remain `OPTIONAL_NOT_CONFIGURED` only outside the current launch gate
- production-provider/write contours stay `FROZEN_EXCLUSION_BEFORE_EXTERNAL_PHASE` after sandbox proof

## External Provider Phase Gate

The external provider phase has started with provider runtime foundation, not fake-green vendor readiness.

Current repo/runtime truth:

- `integration-hub` exposes provider health rows for `diadok`, `smtp_email`, `otp_sms`, `notifications`, and `webhook_intake`.
- `document-service` exposes neutral `esign_provider` health while keeping the existing Provider X-compatible implementation.
- `logistics-service` exposes `logistics_transport` and `osrm_route_compute` provider health.
- `processing-core` admin runtime summary aggregates provider rows and explicitly reports `bank_api_statements`, `erp_1c_delivery`, and `fuel_provider` states.
- Admin Runtime Center renders those provider rows separately from internal service health.

External phase taxonomy:

- selected provider with sandbox proof: `VERIFIED_PROVIDER_SANDBOX`
- selected provider with production proof: `VERIFIED_PROVIDER_PRODUCTION`
- provider intentionally degraded but safe for current environment: `PROVIDER_DEGRADED_OK`
- vendor not selected / adapter not wired: `UNSUPPORTED_PROVIDER_BLOCKER`

Current blocker truth:

- SMS/OTP, bank API, ERP/1C, fuel, logistics, Diadok, and SBIS have sandbox-contract proof without production secrets.
- Production vendor credentials, certificates, and destructive external sync remain `FROZEN_EXCLUSION_BEFORE_EXTERNAL_PHASE`.
- Mailpit/mock modes remain dev/test evidence only and cannot satisfy production external API readiness.

## Backend Harness Exceptions

The internal release gate treats the following as `HARNESS_EXCEPTION_NOT_RUNTIME_BLOCKER` rather than runtime regressions, because live compose health plus mounted smoke already prove the owned contour:

- `billing-clearing`: host-side standalone pytest currently depends on local `celery`; runtime proof remains `scripts/smoke_clearing_batch.cmd`
- `logistics-service`: host-side standalone pytest currently depends on local `fastapi`; runtime proof remains `scripts/smoke_client_logistics.cmd` plus healthy compose service
- `crm-service`: host-side standalone pytest currently depends on local `fastapi` / `sqlalchemy`; runtime proof remains onboarding/partner smokes plus healthy compose service
